Issue #33 · phase (b) · frontend-design

Three directions for the Dashboard primitive library.

These are not progressive refinements — they are three distinct visual languages. Each implements the same seven primitives (Card, StatusPill, ActionButton, ToastRegion, EmptyState, ErrorState, LoadingSkeleton) with the same loading, empty, and error states, so the comparison is on tone, not feature parity. Open each, then pick the one to port into Dioxus.

At a glance

A · Terminal B · Editorial C · HUD
Reads as a power-user TUI an editor's notebook a cockpit panel
Card frame square 1px + ASCII corner no frame, hairline rule frosted glass, notched corners
Headline font IBM Plex Mono · tracked caps Fraunces italic 28–64px Plex Sans Cond. · tracked caps
Numbers tabular · same size as body oversized italic serif headlines JetBrains Mono telemetry readouts
StatusPill (Running) blinking ● + uppercase thin rule + pulsing dot glowing cyan dot, notched outline
ActionButton hover inverts (ink↔bg) arrow nudges right · 200ms floods cyan + glow halo · 120ms
Pending motion blinking caret · 0.6s steps circular ease-out spinner light sweep across surface
Skeleton block ▒ characters, flicker 2.2s left-to-right shimmer scanline sweep, cyan tint
EmptyState glyph ASCII box-drawing oversized italic em-dash glowing diamond outline
ErrorState framing red-bordered panel editor's note · top rule + serif coral cartouche · glow shadow
Best for devs who already live in tmux/k9s a tool you want to take seriously a tool you want to feel powerful
Risk can feel dated if overplayed can feel slow at high information density can drift into AI-cliché if glow is overused
