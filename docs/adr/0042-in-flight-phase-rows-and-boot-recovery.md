# In-flight phase rows and boot-time recovery for orchestrator restarts

Every Codex-driven phase persists a `plan_run_phase_outputs` row with `outcome = 'in_flight'` **before** the **Codex Sandbox** is launched, and updates that row with the parsed phase body on completion. A SIGTERM handler on the control-plane server transitions any still-in-flight rows to `outcome = 'interrupted'` and sends SIGTERM to their captured child PIDs before exiting. On every server boot, a synchronous recovery scanner finds rows left in `in_flight` or `interrupted`, marks their owning **Issue Assignments** `blocked` with new **Block Reason** `OrchestratorRestart`, sweeps each affected **Plan Run** to its `finished` terminal, and emits per-row Activity entries so the dashboard surfaces the recovery as soon as a client reconnects.

The triggering bug was concrete: the control-plane server was stopped while a **Codex Sandbox** was mid-implementation for issue #65; codex finished its work, committed inside the **Assignment Worktree**, and exited. Because no attempt or phase row was persisted at spawn time, the orchestrator had no durable evidence the phase had ever started. The **Issue Assignment** sat at `implementing` indefinitely with no live coordinator to advance it, no recovery path on restart, and no surface for a human to act on. Recovery required hand-editing the database and pushing the agent commit manually.

Pre-spawn persistence covers all four Codex phases uniformly (planning, implementation, review, merge) through the existing `plan_run_phase_outputs` table. The table grows three nullable columns — `process_id INTEGER`, `process_started_at TEXT`, and a widened `outcome` domain that accepts `in_flight` and `interrupted` — plus `body_json` becomes nullable so an in-flight row can exist before the agent returns. The same row is updated on completion with the final outcome and body, so happy-path execution still produces exactly one row per phase. Assignment-scoped rows (implementation, review, merge) carry the `assignment_id` already on the table; Plan Run-scoped rows (planning, merge) carry it as `NULL` as they do today.

The signal handler is `mark-and-kill`, not drain. SIGTERM (and `Ctrl-C`) installs a tokio listener that, on fire, UPDATEs every `in_flight` row to `interrupted`, SIGTERMs each row's captured PID, waits up to two seconds for processes to exit, then returns. Drain mode (waiting for in-flight phases to finish naturally) was rejected because a single implementation phase can run for tens of minutes; blocking shutdown that long is hostile to interactive development. No-op shutdown was rejected because it reproduces the original bug — orphaned codex children write commits the orchestrator never sees. Mark-and-kill makes the loss explicit, durable, and recoverable on the next boot.

Boot recovery runs synchronously after schema migrations and before HTTP routes bind. The scanner reads all rows with `outcome IN ('in_flight', 'interrupted')`, groups them by **Issue Assignment** where applicable, and applies a small fixed table of transitions: an assignment-scoped row in any non-terminal status transitions the **Issue Assignment** to `blocked` with `Block Reason::OrchestratorRestart`; an assignment that had already reached `merge_staged` (push failed but local merge succeeded, per ADR-0037) keeps its staged status because that state is itself recoverable; a planning row with no assignment leaves the **Plan Run** with no claimed work and the run sweeps to `finished` with no merged assignments. The row's own `outcome` is rewritten from `in_flight` to `interrupted` so it is not re-processed on a subsequent boot.

After per-row transitions, the scanner re-evaluates each touched **Plan Run**: if every assignment is now terminal (`merged` / `blocked` / `abandoned`), the **Plan Run** transitions to `finished`. The Plan Run terminal taxonomy stays binary (`running` / `finished`); failure context lives on phase outputs and Activity entries rather than a new `failed` enum value, matching the existing convention where blocked assignments do not turn a Plan Run into a failure state.

Each recovered assignment produces one `ProjectActivity` entry of new kind `AssignmentBlockedOnRestart`, paralleling ADR-0040's pattern of recording state transitions to Activity rather than a dedicated table. The dashboard renders the `OrchestratorRestart` **Block Reason** with the same affordance shape as other typed reasons (badge, recovery hint), and the Activity stream provides the "this is what happened while you were away" trail.

Process identity capture requires refactoring the four Codex phase runners away from `std::process::Command::output()` (which only exposes a PID after the process exits) toward `Command::spawn()` plus an explicit wait, so the orchestrator can record `process_id` and `process_started_at` immediately after spawn. The synchronous-blocking concern (a tokio worker thread held for the duration of a codex run) is left unaddressed here; it is naturally resolved by ADR-0041's `DockerCodexRunner` rewrite, which uses an inherently process-managed `docker run` invocation and can be made async with `tokio::process::Command`. Bridging via `spawn_blocking` in the interim is acceptable but not required by this ADR.

## Considered Options

- **Drain on shutdown — wait for in-flight phases to finish naturally before exit.** Rejected because a single implementation phase can run tens of minutes; blocking `Ctrl-C` that long during development is hostile. Also does not solve the unrelated-crash case (panic, OOM, hardware), which boot recovery must handle regardless.

- **No shutdown handler; rely entirely on boot recovery.** Rejected because it reproduces the original failure mode: the codex child becomes an orphan, finishes its work writing real commits and possibly pushing branches, and exits without the orchestrator ever observing the result. The commit lands but the **Issue Assignment** never advances. Mark-and-kill closes the window deterministically.

- **Adopt in-flight phases on restart — re-attach to the running codex child and resume parsing its stdout.** Rejected because re-attaching to another process's stdout is not portable, mid-flight phase state has no checkpoint anywhere, and ADR-0030 already designs around "phase outputs are durable only at completion." Block-and-recover is honest about the loss and produces a typed **Block Reason** the dashboard surfaces.

- **Pre-spawn write to `assignment_attempts` instead of `plan_run_phase_outputs`.** Rejected for two reasons. First, `assignment_attempts` does not exist for planning and merge phases, which are Plan Run-scoped and would need a parallel table or schema split. Second, `plan_run_phase_outputs` is already the durable record for completion; widening it to also record the spawn moment keeps one source of truth per phase rather than two tables that must be kept in sync.

- **Add a new `failed` terminal to the Plan Run enum so planning-time crashes are visibly distinct from clean finishes.** Rejected because the existing convention already accommodates partial failure — a `finished` Plan Run with all assignments `blocked` is the normal expression of "this run did not produce merged work." Adding a third terminal would force migration of dashboard filters, SSE consumers, and downstream Auto-Replan cycle logic that already classifies by assignment outcome rather than Plan Run terminal.

- **Slice the fix as a single PR.** Rejected because the schema migration + persistence + coordinator pre-spawn write + signal handler are independently shippable from the boot scanner + Activity entry + dashboard surface. Two slices give two independently testable states ("after a kill, the DB has `interrupted` rows" and "after a restart, the dashboard shows the blocked assignments"). Each slice matches the **Issue Assignment** sizing implied by ADR-0028.

- **Make Codex phase runners async in this slice.** Deferred. The sync-blocking concern is real (a tokio worker held for the codex run) but does not block the recovery contract — `spawn_blocking` is an acceptable bridge, and ADR-0041's `DockerCodexRunner` rewrite naturally produces an async surface. Folding async-ification here would expand the slice for marginal correctness benefit.
